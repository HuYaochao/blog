### 二、大模型训练全流程框架

这是理解任何大模型公司的基础认知。

#### 2.1 五个阶段

```
预训练 → 上下文扩展 → 架构改进(续训) → SFT → RL
  ↓           ↓              ↓           ↓      ↓
 基座       能力扩展       效率优化      格式对齐  能力激发
```

#### 2.2 资源分配对比

| 阶段 | V3 (GPU Hours) | V3.2 (估算) | 变化 |
| --- | --- | --- | --- |
| 预训练 | 2,664K (95.6%) | 相同基座 | - |
| 上下文扩展 | 119K (4.3%) | 相同 | - |
| 架构改进（DSA续训） | 0 | ~943B tokens | 新增 |
| 后训练(SFT+RL) | 5K (<1%) | **>10%预训练成本** | **大幅增加** |

**核心变化**：后训练计算量从<1%提升到>10%，这是战略性转变。

#### 2.3 每个阶段在干什么

| 阶段 | 目标 | 典型规模 | 产出 |
| --- | --- | --- | --- |
| 预训练 | 让模型"见过世面" | 万亿tokens级 | 基座模型，能续写文本 |
| 上下文扩展 | 处理长文本 | 数百B tokens | 128K窗口 |
| 架构改进 | 效率优化 | 数百B tokens | 更低推理成本 |
| SFT | 学会"按格式回答" | 百万级样本 | 能对话 |
| RL | 激发能力、对齐价值观 | 数万步训练 | 最终产品 |

### 三、V3.2 三大核心突破

V3.2报告80多页，核心就三件事。

#### 3.1 DeepSeek Sparse Attention (DSA)

**问题**：标准注意力复杂度O(L²)，128K上下文成本太高。

**解决方案**：

- Lightning Indexer：快速判断哪些token重要
- 只选top-k个token做注意力，复杂度降到O(Lk)

**训练流程**（这是关键know-how）：

| 步骤 | 做什么 | 数据量 | 学习率 |
| --- | --- | --- | --- |
| 1. Dense Warm-up | 冻结主模型，只训练indexer | 2.1B tokens (1000步) | 10⁻³ |
| 2. Sparse Training | 全模型适应稀疏模式 | 943.7B tokens (15000步) | 7.3×10⁻⁶ |

**效果**：长上下文推理成本下降50%+，API价格直接砍半。

**理解要点**：架构改动不是从头训，而是在已有checkpoint上"续训"。这是工程智慧——用<1T tokens把效率提上来，而不是重新预训练14.8T。

#### 3.2 可扩展的RL框架

**问题**：想把后训练计算量扩大10倍，但RL训练不稳定。

**解决方案**：GRPO算法 + 四项稳定性改进

| 技术 | 解决什么问题 | 原理（一句话） |
| --- | --- | --- |
| Unbiased KL Estimate | K3估计器在极端情况下梯度爆炸 | 用重要性采样修正偏差 |
| Off-Policy Sequence Masking | 负样本偏离策略太远会误导优化 | 屏蔽KL散度超过阈值的负样本 |
| Keep Routing | MoE专家路由训推不一致 | 保存采样时的路由路径，训练时复用 |
| Keep Sampling Mask | top-p/top-k mask不一致导致重要性采样失效 | 保存采样mask，训练时复用 |

**训练流程**：

```
专家模型训练 → 数据蒸馏 → 混合RL训练
     ↓              ↓            ↓
 6个领域专家    生成领域数据    统一微调
```

六个专家领域：数学、代码、逻辑推理、通用Agent、代码Agent、搜索Agent

**理解要点**：先训专家再蒸馏，比直接混合训练效果更好。这解释了为什么大厂的模型往往"六边形战士"——不是一个模型学所有，而是专家先学好再合并。

#### 3.3 大规模Agent任务合成

**问题**：怎么让模型在调用工具时也能"思考"？

**三步走**：

| 步骤 | 做什么 | 关键点 |
| --- | --- | --- |
| 1. Cold-start | 用prompt模板把推理和工具调用拼在一起 | 模型偶尔能生成正确轨迹，作为RL起点 |
| 2. 上下文管理 | 只在新用户消息时丢弃思考历史 | 避免每次工具调用都重新推理 |
| 3. 大规模合成 | 自动生成环境+任务+验证器 | 1,827环境，85,000+提示词 |

**Agent任务数据来源**：

| 类型 | 数量 | 环境 | 提示词来源 |
| --- | --- | --- | --- |
| Code Agent | 24,667 | 真实GitHub | 提取 |
| Search Agent | 50,275 | 真实搜索API | 合成 |
| General Agent | 4,417 | **合成** | 合成 |
| Code Interpreter | 5,908 | 真实Jupyter | 提取 |

**合成Agent任务的流程**（这是最有价值的know-how）：

```
1. 选任务类别（如：旅行规划）
2. 用bash+search从网上拉数据，存入sandbox数据库
3. 合成工具集（Python函数，只能调用不能直接读库）
4. 从简单任务开始，生成<任务, 解决方案, 验证器>
5. 关键：解决方案只能调用工具，不能直接访问数据库
6. 迭代增加难度，必要时扩展工具集
7. 筛选：只保留pass@100 > 0的任务
```

**核心洞察**：任务难解但易验证（组合爆炸 vs 约束检查）。这跟数学证明一个道理——验证比找答案简单。

**实验验证**：在合成环境上做RL，可以泛化到未见过的真实环境（τ²-bench、MCP-Mark等）。而只在真实代码/搜索环境上训练，反而对这些benchmark没提升。

**结论**：合成数据的多样性比真实性更重要。

### 六、如何判断一家公司是否真的在做模型

这部分是实用指南。

#### 6.1 聊天时的关键问题

| 问题 | 真做模型的人会说 | 假的人可能说 |
| --- | --- | --- |
| 预训练数据怎么处理的？ | 数据清洗pipeline、去重策略、质量过滤器、领域配比调整 | 笼统说"高质量数据"或只谈公开数据集 |
| 训练稳定性怎么保证？ | loss spike处理、learning rate schedule、checkpoint策略 | 不提或说"没遇到问题" |
| RL怎么做的？ | reward设计、KL约束、off-policy问题怎么处理 | 只说"用了RLHF"或"用了DPO" |
| 评估怎么做的？ | 内部benchmark、过拟合检测、人工评估流程 | 只列公开benchmark分数 |
| 遇到过什么坑？ | 具体问题和解决方案 | 说得太顺利或太抽象 |

#### 6.2 报告/宣传中的真假信号

**真的迹象**：

- 披露负面信息（V3.2报告承认token效率不如Gemini）
- 具体数字（GPU hours、tokens数量、训练步数）
- 失败和trade-off（为控制成本加了长度限制）
- 消融实验（单独拿掉某个技术看效果变化）
- 开源可复现

**假的迹象**：

- 只报好的benchmark，不报差的
- 笼统的"创新架构"但没有技术细节
- 声称全面超越但没有对比细节
- 训练成本和数据规模不透明
- 没有消融实验

#### 6.3 深度探针问题

如果想测试对方是否真懂：

| 领域 | 探针问题 | 期望回答方向 |
| --- | --- | --- |
| MoE | 负载均衡怎么做的？ | auxiliary loss vs loss-free的trade-off |
| 长上下文 | 128K是怎么扩上去的？ | RoPE外推、位置编码调整、长文本数据配比 |
| RL | GRPO和PPO的区别？ | 不需要critic模型，计算效率高 |
| 推理能力 | R1的思考是怎么来的？ | 纯RL（R1-Zero）vs 冷启动+RL（R1） |
| Agent | 工具调用时思考怎么保留？ | 上下文管理策略、token效率问题 |

#### 6.4 快速判断框架

```
第一层：有没有做过从头训练（pretrain）？
  └─ 没有 → 大概率是微调/套壳
  └─ 有 → 进入第二层

第二层：训练规模多大？
  └─ <100B tokens → 小规模实验
  └─ 100B-1T tokens → 中等规模
  └─ >1T tokens → 正经做模型

第三层：后训练做了什么？
  └─ 只有SFT → 基础水平
  └─ SFT + 简单RL → 中等水平
  └─ SFT + 可扩展RL + 多领域 → 前沿水平

第四层：有没有独特创新？
  └─ 纯复现 → 工程能力
  └─ 有改进但增量 → 有研究能力
  └─ 有原创方法被验证有效 → 真正的研究团队
```

### 七、关键洞察总结

#### 7.1 技术趋势

1. **后训练计算量大幅增加**：从<1%到>10%，RL成为核心战场
2. **架构改进通过续训实现**：不需要重新预训练，用<1T tokens改架构
3. **合成数据比真实数据更有效**：多样性>真实性
4. **专家蒸馏成为标配**：先训专家，再合并
5. **Token效率是新瓶颈**：准确率够了，但推理成本太高

#### 7.2 DeepSeek的策略

1. **开源换信任**：详细披露方法论，建立社区信任
2. **成本优先**：每个技术决策都考虑效率
3. **增量迭代**：V3→V3.1→V3.2稳步推进，不追求一步到位
4. **诚实披露局限**：主动说token效率不如竞品

#### 7.3 对从业者的启示

1. 后训练是差异化的关键——预训练趋同，后训练定高下
2. 合成数据能力是核心竞争力——谁能大规模合成高质量数据谁赢
3. RL稳定性是扩展的前提——想扩10倍计算量，必须先解决稳定性
4. Agent是下一个战场——V3.2明确在这个方向发力
