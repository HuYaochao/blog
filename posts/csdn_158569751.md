简单说一下：

**33K、44K** 是文件大小，33K = 33KB，44K = 44KB。说明这两个文件内容很多——SKILL.md 有近 500 行指令，viewer.html 是一个功能完整的单页 Web 应用。

---

**整体分三层理解：**

**核心（你只需要这个）**

| 文件 | 干嘛的 |
| --- | --- |
| `SKILL.md` | 主指南，告诉 AI "怎么帮用户创建技能"的全部流程和规则 |

**评估系统（用来测试技能好不好）**

| 文件 | 干嘛的 |
| --- | --- |
| `agents/grader.md` | 告诉 AI 子代理怎么给测试结果打分（这个断言过了没？） |
| `agents/comparator.md` | 告诉 AI 子代理怎么做盲测——给它两个输出，不说哪个是新版哪个是旧版，让它判断哪个好 |
| `agents/analyzer.md` | 告诉 AI 子代理怎么分析基准数据——找出哪些测试不稳定、哪些断言没区分度 |
| `assets/eval_review.html` | 一个网页模板，让用户审查触发评估查询（勾选哪些该触发、哪些不该） |
| `eval-viewer/viewer.html` | 44K 的大网页，用来展示测试结果——左边看输出，右边写反馈，还有基准对比图表 |
| `eval-viewer/generate_review.py` | 把测试结果数据注入到 viewer.html 里，然后在浏览器中打开 |
| `references/schemas.md` | 定义各种 JSON 文件的格式规范（evals.json 长什么样、grading.json 长什么样） |

**脚本工具（自动化干活的）**

| 脚本 | 干嘛的 |
| --- | --- |
| `quick_validate.py` | 检查 SKILL.md 格式对不对（有没有 name、description 等） |
| `package_skill.py` | 把技能文件夹打包成 `.skill` 文件 |
| `run_eval.py` | 拿测试提示词跑一遍技能，保存输出结果 |
| `aggregate_benchmark.py` | 把多个测试结果汇总成一份基准报告（通过率、耗时、token 数） |
| `generate_report.py` | 生成人可读的评估报告 |
| `improve_description.py` | 让 AI 根据触发失败的案例，自动改写 description |
| `run_loop.py` | 把"评估→改写→再评估"串成自动循环，跑 5 轮选最优 description |
| `utils.py` | 上面几个脚本共用的工具函数 |

---

**一句话总结**：SKILL.md 是大脑，agents 是评审团，eval-viewer 是结果展示台，scripts 是流水线工人。你只想做技能的话，只需要关心 SKILL.md 的写法就够了，其余都是"工业级质量保证"用的。

## 一、Claude技能创建器指南\_SKILL中文版

---

name: skill-creator  
description: 创建新技能、修改和改进现有技能、衡量技能表现。当用户想要从零创建技能、编辑或优化现有技能、运行评估来测试技能、通过方差分析进行基准测试、或优化技能描述以提高触发准确率时使用。

---

## 技能创建器

一个用于创建新技能并迭代改进的技能。

整体流程大致如下：

- 确定你想让技能做什么，以及大致的实现方式
- 编写技能草稿
- 创建几个测试提示词，并让"拥有该技能的 Claude"运行它们
- 帮助用户从定性和定量两个角度评估结果
  - 在后台运行测试的同时，如果还没有定量评估标准就起草一些（如果已有，可以直接使用或根据需要修改）。然后向用户解释这些评估标准
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果供其查看，同时让他们看到定量指标
- 根据用户对结果的评估反馈（以及定量基准测试中暴露的明显问题）重写技能
- 反复迭代直到满意
- 扩大测试集，在更大范围内再次测试

你在使用此技能时的工作是：判断用户处于流程的哪个阶段，然后帮助他们推进。比如，用户可能说"我想做一个关于 X 的技能"。你可以帮助他们明确需求、编写草稿、编写测试用例、确定评估方式、运行所有提示词，然后迭代。

另一方面，用户可能已经有了技能草稿。这种情况下你可以直接进入评估/迭代循环。

当然，你应该始终保持灵活——如果用户说"我不需要跑一堆评估，跟我随意聊就行"，你也可以这样做。

技能完成后（顺序同样灵活），你还可以运行技能描述优化器（我们有专门的脚本来做这件事）来优化技能的触发效果。

明白了？好的。

### 与用户沟通

技能创建器的使用者可能跨越很大的技术背景范围。现在有一个趋势——Claude 的强大能力正在激励各行各业的人打开终端，父母和祖父母也开始搜索"怎么安装 npm"。当然，大部分用户可能还是有一定技术基础的。

所以请注意语境线索，理解如何措辞你的沟通！在默认情况下，给你一些参考：

- "评估"和"基准测试"属于边界词汇，但可以用
- "JSON"和"断言"这类词，你需要看到用户确实了解这些概念的明确信号后才能直接使用，否则需要解释

如果你不确定用户是否理解，简短地解释一下术语是没问题的。

### 创建技能

#### 捕捉意图

首先理解用户的意图。当前对话可能已经包含用户想要固化的工作流（例如他们说"把这个变成一个技能"）。如果是这样，先从对话历史中提取答案——使用了哪些工具、步骤顺序、用户做了哪些修正、观察到的输入/输出格式。用户可能需要补充一些缺失信息，并在进入下一步之前确认。

1. 这个技能应该让 Claude 做什么？
2. 什么时候应该触发这个技能？（什么样的用户表述/场景）
3. 期望的输出格式是什么？
4. 是否需要设置测试用例来验证技能？输出可客观验证的技能（文件转换、数据提取、代码生成、固定工作流步骤）适合用测试用例；输出主观的技能（写作风格、艺术创作）通常不需要。根据技能类型建议合适的默认方案，但让用户决定。

#### 访谈与调研

主动询问关于边界情况、输入/输出格式、示例文件、成功标准和依赖项的问题。在这些问题搞清楚之前不要急于编写测试提示词。

检查可用的 MCP——如果有助于调研（搜索文档、查找类似技能、查找最佳实践），可以通过子代理并行调研（如果可用），否则内联调研。准备好背景知识以减轻用户的负担。

#### 编写 SKILL.md

根据用户访谈，填充以下组件：

- **name**（名称）：技能标识符
- **description**（描述）：何时触发、做什么。这是主要的触发机制——既要包含技能的功能，也要包含具体的使用场景。所有"何时使用"的信息都放在这里，而不是正文中。注意：目前 Claude 有一种倾向是"欠触发"技能——在技能有用时却不使用。为了对抗这一点，请让技能描述稍微"强势"一些。例如，不要写"如何构建一个简单快速的仪表盘来展示内部 Anthropic 数据"，而是写"如何构建一个简单快速的仪表盘来展示内部 Anthropic 数据。当用户提到仪表盘、数据可视化、内部指标、或想要展示任何类型的公司数据时，务必使用此技能，即使他们没有明确要求’仪表盘’。"
- **compatibility**（兼容性）：所需工具、依赖项（可选，很少需要）
- **技能正文 😃**

#### 技能编写指南

##### 技能的结构

```
skill-name/
├── SKILL.md（必需）
│   ├── YAML 前置元数据（name、description 必填）
│   └── Markdown 指令
└── 捆绑资源（可选）
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 按需加载到上下文中的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

##### 渐进式加载

技能使用三级加载系统：

1. **元数据**（名称 + 描述）- 始终在上下文中（约 100 词）
2. **SKILL.md 正文** - 技能触发时进入上下文（理想情况下 <500 行）
3. **捆绑资源** - 按需加载（无限制，脚本可在不加载的情况下执行）

这些字数是近似值，如果需要可以更长。

**关键模式：**

- SKILL.md 保持在 500 行以内；如果接近此限制，添加额外的层级结构，并清楚地指示使用该技能的模型接下来应该去哪里查看更多内容
- 从 SKILL.md 中清晰地引用文件，并说明何时需要读取它们
- 对于大型参考文件（>300 行），包含目录

**领域组织**：当一个技能支持多个领域/框架时，按变体组织：

```
cloud-deploy/
├── SKILL.md（工作流 + 选择逻辑）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

Claude 只读取相关的参考文件。

##### 无意外原则

不用说，技能不得包含恶意软件、漏洞利用代码或任何可能危害系统安全的内容。技能的内容在描述其意图时不应让用户感到意外。不要配合创建误导性技能或旨在促进未授权访问、数据窃取或其他恶意活动的技能请求。不过像"扮演某个角色"之类的是可以的。

##### 编写模式

指令优先使用祈使句形式。

**定义输出格式** - 你可以这样做：

```
## 报告结构
始终使用这个精确模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式** - 包含示例很有用。你可以这样格式化（但如果示例中有"输入"和"输出"你可能需要稍作调整）：

```
## 提交信息格式
**示例 1：**
输入：添加了使用 JWT 令牌的用户认证
输出：feat(auth): implement JWT-based authentication
```

#### 编写风格

尝试向模型解释为什么某些事情很重要，而不是使用大量严厉的"必须"。运用心智理论（Theory of Mind），尽量让技能通用化，不要过度局限于特定示例。先写一个草稿，然后以全新的眼光审视并改进它。

#### 测试用例

编写技能草稿后，想出 2-3 个真实的测试提示词——真正用户实际会说的那种。与用户分享："这是我想尝试的几个测试用例。看起来可以吗，还是你想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。先不要写断言——只要提示词。你会在下一步中在运行进行时起草断言。

```
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示词",
      "expected_output": "预期结果的描述",
      "files": []
    }
  ]
}
```

参见 `references/schemas.md` 获取完整的 schema（包括 `assertions` 字段，你稍后会添加）。

### 运行和评估测试用例

本节是一个连续的序列——不要中途停下。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 目录中，作为技能目录的同级目录。在工作区中，按迭代组织结果（`iteration-1/`、`iteration-2/` 等），每个测试用例在其中有一个目录（`eval-0/`、`eval-1/` 等）。不要预先创建所有目录——在进行过程中创建。

#### 步骤 1：在同一轮中启动所有运行（含技能版和基线版）

对于每个测试用例，在同一轮中启动两个子代理——一个使用技能，一个不使用。这很重要：不要先启动含技能版的运行，之后再回来做基线版。同时启动所有运行，这样它们会大约同时完成。

**含技能版运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval files if any, or "none">
- 输出保存到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 需要保存的输出：<用户关心的内容——如"docx 文件"、"最终的 CSV">
```

**基线版运行**（相同提示词，但基线取决于上下文）：

- **创建新技能**：完全不用技能。相同提示词，不指定技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：使用旧版本。编辑前先快照技能（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例编写 `eval_metadata.json`（断言可以暂时为空）。给每个评估一个基于测试内容的描述性名称——不要只用"eval-0"。目录名也使用这个名称。如果这次迭代使用了新的或修改过的评估提示词，为每个新的评估目录创建这些文件——不要假设它们从之前的迭代延续。

```
{
  "eval_id": 0,
  "eval_name": "此处填描述性名称",
  "prompt": "用户的任务提示词",
  "assertions": []
}
```

#### 步骤 2：运行进行中时，起草断言

不要干等着运行完成——你可以利用这段时间高效工作。为每个测试用例起草定量断言，并向用户解释它们。如果 `evals/evals.json` 中已有断言，审查它们并解释它们检查的内容。

好的断言是客观可验证的，且有描述性的名称——在基准测试查看器中应该能一目了然，让人立刻理解每个断言检查的内容。主观性技能（写作风格、设计质量）更适合定性评估——不要在需要人类判断的东西上强行使用断言。

起草完成后，更新 `eval_metadata.json` 文件和 `evals/evals.json` 中的断言。同时向用户解释他们将在查看器中看到什么——包括定性输出和定量基准。

#### 步骤 3：运行完成时，捕获计时数据

当每个子代理任务完成时，你会收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传递，不会在其他地方持久化。在每个通知到达时立即处理，而不是尝试批量处理。

#### 步骤 4：评分、汇总并启动查看器

所有运行完成后：

1. **为每次运行评分** — 启动一个评分子代理（或内联评分），读取 `agents/grader.md` 并根据输出评估每个断言。将结果保存到每个运行目录中的 `grading.json`。grading.json 中的 expectations 数组必须使用 `text`、`passed` 和 `evidence` 字段（不是 `name`/`met`/`details` 或其他变体）——查看器依赖这些确切的字段名。对于可以通过程序检查的断言，编写并运行脚本而不是用眼睛看——脚本更快、更可靠，且可以在迭代间复用。
2. **汇总为基准测试** — 从 skill-creator 目录运行汇总脚本：

   ```
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <n>
   ```

   这会生成 `benchmark.json` 和 `benchmark.md`，包含每个配置的通过率、时间和 token 数，带有均值 ± 标准差和差值。如果手动生成 benchmark.json，参见 `references/schemas.md` 获取查看器期望的确切 schema。  
   将每个 with\_skill 版本放在其基线对应版本之前。
3. **进行分析** — 读取基准测试数据并发现汇总统计可能隐藏的模式。参见 `agents/analyzer.md`（"分析基准测试结果"部分）了解要查找什么——如无论是否使用技能都总是通过的断言（无区分度）、高方差评估（可能不稳定）、以及时间/token 的权衡。
4. **启动查看器**，同时展示定性输出和定量数据：

   ```
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```

   对于第 2 次及以后的迭代，还要传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **Cowork / 无界面环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，使用 `--static <output_path>` 写一个独立的 HTML 文件而不是启动服务器。当用户点击"提交所有评审"时，反馈会作为 `feedback.json` 文件下载。下载后，将 `feedback.json` 复制到工作区目录中供下次迭代使用。

注意：请使用 generate\_review.py 来创建查看器；不需要自己写 HTML。

5. **告诉用户**类似这样的话：“我已经在浏览器中打开了结果。有两个标签页——'输出’可以逐个查看每个测试用例并留下反馈，'基准测试’显示定量对比。看完后回来告诉我。”

#### 用户在查看器中看到什么

"输出"标签页一次显示一个测试用例：

- **提示词**：给出的任务
- **输出**：技能生成的文件，尽可能内联渲染
- **上一次输出**（第 2 次迭代起）：折叠部分，显示上次迭代的输出
- **正式评分**（如果运行了评分）：折叠部分，显示断言的通过/失败
- **反馈**：一个文本框，输入时自动保存
- **上一次反馈**（第 2 次迭代起）：他们上次的评论，显示在文本框下方

"基准测试"标签页显示统计摘要：每个配置的通过率、时间和 token 使用量，带有每个评估的明细和分析师观察。

导航通过上一个/下一个按钮或方向键。完成后点击"提交所有评审"，将所有反馈保存到 `feedback.json`。

#### 步骤 5：读取反馈

当用户告诉你他们完成了，读取 `feedback.json`：

```
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少坐标轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，很喜欢", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈表示用户认为没问题。将改进重点放在用户有具体意见的测试用例上。

完成后关闭查看器服务器：

```
kill $VIEWER_PID 2>/dev/null
```

### 改进技能

这是循环的核心。你已经运行了测试用例，用户已经审查了结果，现在你需要根据他们的反馈让技能变得更好。

#### 如何思考改进

1. **从反馈中归纳**。这里的大局是：我们在尝试创建可以被使用数百万次（也许确实如此，甚至更多谁知道呢）的技能，跨越许多不同的提示词。在这里，你和用户只是在几个例子上反复迭代，因为这样更快。用户对这些例子了如指掌，可以快速评估新的输出。但如果你和用户共同开发的技能只对这些例子有效，那就没用了。与其加入琐碎的过拟合修改或压迫性的严格"必须"，如果有某个顽固的问题，你可以尝试换一种思路，使用不同的比喻或推荐不同的工作模式。尝试的成本相对较低，也许你会发现很棒的方案。
2. **保持提示词精简**。移除没有发挥作用的内容。确保阅读运行日志，而不仅仅是最终输出——如果看起来技能在让模型浪费大量时间做无用功，你可以尝试去掉技能中导致这种情况的部分，看看效果如何。
3. **解释为什么**。努力解释你要求模型做每件事的**原因**。现在的大语言模型很*聪明*。它们有良好的心智理论，在好的框架下可以超越机械式的指令，真正产生出色的成果。即使用户的反馈简短或带有挫败感，也要试着真正理解任务、理解用户为什么写下那些反馈、以及他们到底写了什么，然后将这种理解传递到指令中。如果你发现自己在写全大写的"始终"或"绝不"，或使用超级严格的结构，那是一个黄色警告——如果可能的话，重新表述并解释推理，让模型理解你要求的事情为什么重要。这是一种更人性化、更强大、更有效的方法。
4. **寻找测试用例间的重复工作**。阅读测试运行的日志，注意子代理是否都独立地编写了类似的辅助脚本或采取了相同的多步骤方法。如果所有 3 个测试用例都导致子代理编写了 `create_docx.py` 或 `build_chart.py`，这是一个强烈信号——技能应该捆绑这个脚本。写一次，放在 `scripts/` 中，告诉技能使用它。这可以让未来的每次调用不必重新发明轮子。

这个任务相当重要（我们试图在这里创造每年数十亿美元的经济价值！），你的思考时间不是瓶颈；慢慢来，好好琢磨。建议先写一个修改草稿，然后以全新的眼光审视并进行改进。真正尽力去理解用户的思维，理解他们想要和需要什么。

#### 迭代循环

改进技能后：

1. 将你的改进应用到技能
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录中，包括基线运行。如果你在创建新技能，基线始终是 `without_skill`（无技能）——这在迭代间保持不变。如果你在改进现有技能，根据你的判断决定什么作为合适的基线：用户最初带来的原始版本，还是前一次迭代。
3. 启动查看器，使用 `--previous-workspace` 指向前一次迭代
4. 等待用户审查并告诉你他们完成了
5. 读取新的反馈，再次改进，重复

持续进行直到：

- 用户说他们满意了
- 反馈全部为空（一切看起来都好）
- 你没有取得有意义的进展

### 进阶：盲测对比

对于需要更严格地比较两个技能版本的情况（例如用户问"新版本真的更好吗？"），有一个盲测对比系统。阅读 `agents/comparator.md` 和 `agents/analyzer.md` 了解详情。基本思路是：将两个输出交给一个独立代理，不告诉它哪个是哪个，让它判断质量。然后分析获胜者为什么赢了。

这是可选的，需要子代理，大多数用户不需要它。人工审查循环通常就足够了。

### 描述优化

SKILL.md 前置元数据中的 description 字段是决定 Claude 是否调用技能的主要机制。创建或改进技能后，主动提出优化描述以提高触发准确率。

#### 步骤 1：生成触发评估查询

创建 20 个评估查询——混合应该触发和不应该触发的。保存为 JSON：

```
[
  {"query": "用户提示词", "should_trigger": true},
  {"query": "另一个提示词", "should_trigger": false}
]
```

查询必须是真实的，是 Claude Code 或 Claude.ai 用户实际会输入的内容。不要抽象的请求，而是具体的、详细的请求。例如，文件路径、关于用户工作或情况的个人背景、列名和值、公司名称、URL。一些背景故事。有些可能是小写的、包含缩写或拼写错误或口语化表达。使用不同长度的混合，重点放在边界情况上而不是清晰明确的情况（用户会有机会签署确认它们）。

不好的例子：`"格式化这些数据"`、`"从 PDF 中提取文本"`、`"创建一个图表"`

好的例子：`"我老板刚发了这个 xlsx 文件给我（在我的下载文件夹里，好像叫什么 'Q4 销售 最终版 终终终版 v2.xlsx'），她要我加一列显示利润率百分比。收入在 C 列，成本在 D 列吧应该是"`

对于**应该触发**的查询（8-10 个），考虑覆盖面。你需要同一意图的不同表述——有些正式，有些随意。包含用户没有明确说出技能名称或文件类型但明显需要它的情况。加入一些不常见的用例和该技能与其他技能竞争但应该胜出的情况。

对于**不应该触发**的查询（8-10 个），最有价值的是"擦边"情况——与技能共享关键词或概念但实际上需要不同东西的查询。思考相邻领域、天真的关键词匹配会触发但不应该触发的模糊表述，以及查询涉及技能所做的事情但在另一个工具更合适的上下文中的情况。

关键要避免的：不要让不应该触发的查询明显无关。"写一个斐波那契函数"作为 PDF 技能的反面测试太简单了——它不测试任何东西。反面情况应该是真正棘手的。

#### 步骤 2：与用户审查

使用 HTML 模板向用户展示评估集以供审查：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的 JSON 数组（不要加引号——它是一个 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能的名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能的当前描述
3. 写入临时文件（如 `/tmp/eval_review_<skill-name>.html`）并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换是否应触发、添加/删除条目，然后点击"导出评估集"
5. 文件下载到 `~/Downloads/eval_set.json`——检查下载文件夹中最新版本以防有多个（如 `eval_set (1).json`）

这一步很重要——糟糕的评估查询会导致糟糕的描述。

#### 步骤 3：运行优化循环

告诉用户：“这会需要一些时间——我会在后台运行优化循环并定期检查进度。”

将评估集保存到工作区，然后在后台运行：

```
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用你系统提示中的模型 ID（驱动当前会话的那个），这样触发测试与用户实际体验一致。

在运行期间，定期查看输出以向用户更新当前在哪次迭代以及分数如何。

这会自动处理完整的优化循环。它将评估集分为 60% 训练集和 40% 留出测试集，评估当前描述（每个查询运行 3 次以获得可靠的触发率），然后调用 Claude 使用扩展思考来根据失败的情况提出改进建议。它在训练集和测试集上重新评估每个新描述，最多迭代 5 次。完成后，它会在浏览器中打开一个 HTML 报告，显示每次迭代的结果，并返回包含 `best_description` 的 JSON——根据测试分数而非训练分数选择，以避免过拟合。

#### 技能触发的工作原理

理解触发机制有助于设计更好的评估查询。技能以名称 + 描述的形式出现在 Claude 的 `available_skills` 列表中，Claude 根据该描述决定是否查阅技能。重要的是要知道，Claude 只会在无法轻松独立处理的任务上查阅技能——简单的、单步骤的查询如"读取这个 PDF"可能不会触发技能，即使描述完美匹配，因为 Claude 可以用基本工具直接处理。复杂的、多步骤的或专业化的查询在描述匹配时会可靠地触发技能。

这意味着你的评估查询应该足够实质性，以至于 Claude 确实会从查阅技能中受益。简单的查询如"读取文件 X"是糟糕的测试用例——无论描述质量如何，它们都不会触发技能。

#### 步骤 4：应用结果

从 JSON 输出中取 `best_description`，更新技能 SKILL.md 的前置元数据。向用户展示修改前后的对比并报告分数。

---

#### 打包并交付（仅在有 `present_files` 工具时）

检查你是否有 `present_files` 工具。如果没有，跳过此步骤。如果有，打包技能并向用户呈现 .skill 文件：

```
python -m scripts.package_skill <path/to/skill-folder>
```

打包后，引导用户到生成的 `.skill` 文件路径，以便他们安装。

### Claude.ai 特定说明

在 Claude.ai 中，核心工作流相同（草稿 → 测试 → 审查 → 改进 → 重复），但由于 Claude.ai 没有子代理，某些机制会有所变化。以下是需要调整的部分：

**运行测试用例**：没有子代理意味着不能并行执行。对于每个测试用例，读取技能的 SKILL.md，然后按照其指令自己完成测试提示词。逐个进行。这不如独立子代理严格（你写了技能又在运行它，所以你有完整的上下文），但这是一个有用的健全性检查——人工审查步骤可以弥补。跳过基线运行——只使用技能完成请求的任务。

**审查结果**：如果你无法打开浏览器（例如 Claude.ai 的虚拟机没有显示器，或你在远程服务器上），完全跳过浏览器查看器。改为直接在对话中展示结果。对于每个测试用例，显示提示词和输出。如果输出是用户需要看的文件（如 .docx 或 .xlsx），保存到文件系统并告诉他们位置以便下载查看。内联询问反馈：“看起来怎么样？有什么需要改的吗？”

**基准测试**：跳过定量基准测试——它依赖于没有子代理时无意义的基线对比。专注于用户的定性反馈。

**迭代循环**：与之前相同——改进技能，重新运行测试用例，询问反馈——只是中间没有浏览器查看器。如果你有文件系统，仍然可以将结果组织到迭代目录中。

**描述优化**：此部分需要 `claude` CLI 工具（具体来说是 `claude -p`），这只在 Claude Code 中可用。如果在 Claude.ai 上则跳过。

**盲测对比**：需要子代理。跳过。

**打包**：`package_skill.py` 脚本在任何有 Python 和文件系统的环境中都能工作。在 Claude.ai 上，你可以运行它，用户可以下载生成的 `.skill` 文件。

**更新现有技能**：用户可能要求你更新现有技能，而不是创建新的。在这种情况下：

- **保留原始名称。** 记下技能的目录名和 `name` 前置元数据字段——原样使用。例如，如果安装的技能是 `research-helper`，输出 `research-helper.skill`（不是 `research-helper-v2`）。
- **编辑前复制到可写位置。** 安装的技能路径可能是只读的。复制到 `/tmp/skill-name/`，在那里编辑，然后从副本打包。
- **如果手动打包，先暂存到 `/tmp/`**，然后复制到输出目录——由于权限问题直接写入可能失败。

## 二、简要版（只要md）

## 技能创建器指南（精简版）

> 只需一个 SKILL.md 就能创建技能。本文档涵盖从编写到打包的全部要点。

---

### 一、技能是什么

技能就是一个 `SKILL.md` 文件，放在一个文件夹里。AI 根据文件头部的 `description` 决定是否使用它。

最简结构：

```
my-skill/
└── SKILL.md
```

需要的话可以加子目录（`scripts/`、`references/`、`assets/`），但不是必须的。

---

### 二、SKILL.md 格式

```
---
name: my-skill
description: 这里写触发条件和功能描述。AI 靠这段话决定是否调用技能，所以要写得具体、稍微"强势"一些，覆盖用户可能的各种说法。
---

# 技能标题

## 概述
做什么、为什么需要

## 工作流程
### 步骤 1：...
### 步骤 2：...

## 输出格式
定义模板或示例

## 注意事项
边界情况、限制
```

#### 前置元数据（必填）

| 字段 | 说明 |
| --- | --- |
| `name` | 英文标识符，小写+连字符，如 `data-analysis` |
| `description` | **最关键的字段**。决定触发时机。要写明：技能做什么 + 什么时候该用 |

#### description 怎么写好

AI 倾向于"欠触发"，所以描述要主动一些：

❌ 不好：`分析数据报表`

✅ 好：`数据报表分析与洞察报告生成。当用户提到"数据分析"、"报表分析"、"趋势分析"、"指标异常"等内容时，务必触发此技能，即使用户没有明确说"生成报告"。`

#### 正文怎么写

- **祈使句**：直接说"做什么"，不要"你应该做什么"
- **解释为什么**：比堆砌"必须""绝不"更有效，AI 理解原因后表现更好
- **给示例**：输入→输出的具体例子比抽象规则有用
- **保持通用**：不要过度针对特定案例，要对广泛输入都有效
- **控制长度**：正文建议 500 行以内，超出的拆到 `references/` 子目录

---

### 三、创建流程

#### 1. 明确需求

问自己四个问题：

- 技能让 AI 做什么？
- 什么时候触发？（用户会怎么说）
- 输出什么格式？
- 有没有边界情况？

#### 2. 写 SKILL.md

按上面的格式写。先写草稿，再审视改进。

#### 3. 验证（可选）

手动检查：

- 有 `---` 包裹的 YAML 头？
- `name` 和 `description` 都有值？
- 正文有清晰的章节结构？
- 没有未替换的占位符 `{{...}}`？

#### 4. 测试

想 2-3 个真实用户会说的话，试试技能效果。看看输出是否符合预期。不满意就改 SKILL.md，再试，循环直到满意。

#### 5. 打包

技能文件夹直接 zip 压缩，改后缀为 `.skill` 即可：

```
zip -r my-skill.skill my-skill/
```

### 四、完整示例

以下是一个"会议纪要生成"技能的完整 SKILL.md：

```
---
name: meeting-notes
description: 会议纪要生成技能。当用户提到"会议纪要"、"会议记录"、"会议总结"、"整理会议内容"、上传了会议录音转写文本、或描述了一场会议希望整理时，务必使用此技能。
---

# 会议纪要生成

## 概述

将会议内容（对话描述或上传的转写文本）整理为结构化的会议纪要。

## 工作流程

### 步骤 1：获取会议信息

如果用户上传了文件，读取文件内容。
如果用户口头描述，提取关键信息：会议主题、参与人、时间、讨论要点。
信息不足时追问，但每次不超过 2 个问题。

### 步骤 2：生成纪要

按以下模板生成 Markdown 文件：

# 会议纪要

- **会议主题**：
- **时间**：
- **参与人**：

## 讨论要点
1. ...

## 决议事项
| 事项 | 负责人 | 截止日期 |
|------|--------|----------|

## 待跟进
- ...

### 步骤 3：输出

保存为 `会议纪要_[主题]_[日期].md`，交付给用户。

## 注意事项

- 保持客观，不添加会议中没有的内容
- 如果信息模糊，用"（待确认）"标注
```

### 五、常见问题

**Q：技能可以包含脚本吗？**  
可以，放在 `scripts/` 目录下。但大多数技能只要一个 SKILL.md 就够了。

**Q：description 多长合适？**  
50-300 字符。太短触发不到，太长浪费上下文。

**Q：怎么知道技能有没有被触发？**  
如果 AI 的回答明显遵循了你技能中定义的流程和格式，说明触发了。否则可以在 description 中加更多触发关键词。

**Q：可以有多个 SKILL.md 吗？**  
不行，一个技能文件夹只能有一个 SKILL.md。但可以用 `references/` 放补充文档。

**Q：技能之间会冲突吗？**  
可能会。如果两个技能的 description 关键词重叠，AI 可能选错。解决方法是让 description 更精确地区分使用场景。
