> 核心命题：无论哪种模式，20步连续任务都会撞上注意力稀释的墙。区别只在于**怎么撞**和**撞了之后还剩什么**。

---

### 一、硬约束：两个不可放弃的物理事实

**事实1**：上下文窗口是固定的。200k tokens也好，1M tokens也好，都是一个有限的数字。窗口翻5倍不等于问题消失——只是把撞墙的时间推迟了，注意力衰减的物理规律不会因为窗口变大而改变。

**事实2**：注意力权重不是均匀分布的。开头和最近几轮权重高，中间的内容权重衰减。窗口越满，衰减越严重。

由此推出：**任何20步连续任务，都是在和这两个事实做博弈。**

---

### 二、前置知识：大模型是怎么"走"多步的

在讨论两种模式的优劣之前，必须先理解一个反直觉的事实：**大模型没有记忆，每一步都是从零开始。**

#### 2.1 每次推理都是"一次性阅读理解"

大模型不像人类那样"经历"了步骤1、2、3之后，脑子里自然留下了印象。它的工作方式更像是：每次需要输出时，系统把**所有历史内容打包成一份文档**，模型从头读一遍这份文档，然后基于阅读理解给出回答。

以一个三步任务为例——读取文件A，读取文件B，基于两者做分析：

**步骤1**：模型收到的输入是 `[任务描述]`，输出对A的分析。

**步骤2**：模型收到的输入是 `[任务描述] + [A的内容] + [步骤1的输出] + [B的内容]`，它重新"读"一遍所有内容，输出对A和B的综合分析。

**步骤3**：模型收到的输入是 `[任务描述] + [A的内容] + [步骤1的输出] + [B的内容] + [步骤2的输出] + [新指令]`，它再次从头"读"一遍全部内容，然后输出。

关键在于：**步骤3的模型和步骤1的模型是同一个模型，没有任何内部状态被保留**。它不是"记得"之前分析了A和B，而是在步骤3的上下文里重新看到了A和B的内容以及之前的分析文本。

#### 2.2 所谓"多步推理"，本质是上下文堆叠

这意味着什么？意味着所谓的"20步连续任务"，本质上是**一份不断膨胀的文档被反复从头阅读**。每多走一步，这份文档就多一层——上一步读取的内容、上一步的思考输出、新的指令、新读取的内容……全部累加。

所有围绕大模型的工程——对话管理、compact压缩、子代理、进度文件——**都是在这份不断膨胀的文档上做文章**：怎么控制它的大小、怎么决定哪些内容留下哪些压缩、怎么保证关键信息始终处在模型注意力的高权重区域。

理解了这一点，后面的分析就有了基础：两种模式的所有优劣，归根到底都是在回答同一个问题——**怎么在一份有限大小的文档里，装下足够多的有效信息，同时不让关键信息被淹没**。

### 三、两种工具加载模式的本质区别

#### 3.1 Load tools when needed（按需加载）

工具定义不预先占用上下文。需要某个工具时才加载其schema，用完后在compact时可以被压缩掉。

**本质**：用"每次加载的延迟"换"更大的可用空间"。

#### 3.2 Tools already loaded（预加载）

所有工具定义常驻上下文。随时可调用，但每轮推理都在为这些定义付出注意力成本。

**本质**：用"持续的空间占用"换"随时可用的便利"。

#### 3.3 空间账本对比

假设一个典型场景：可用窗口200k tokens，工具定义合计约15k-25k tokens。

| 项目 | 按需加载 | 预加载 |
| --- | --- | --- |
| 工具定义占用 | 仅当前步骤需要的（约2k-5k） | 全量常驻（约15k-25k） |
| 基线可用空间 | ~195k | ~175k |
| compact触发频率 | 较低 | 较高 |
| 每次compact释放的空间 | 较多（工具定义也可清除） | 较少（工具定义不可清除） |

按需加载在20步任务中是更优的选择——这20k的差距不是小事，它直接决定能多撑几步才触发第一次compact。

### 四、注意力衰减 + compact压缩：20步任务的双重绞杀

多步骤任务中有两个力量同时作用，理解它们就能理解两种模式为什么各有各的崩法。

#### 4.1 注意力衰减（步骤越多越严重）

Transformer的注意力机制对上下文中所有token做加权投票。随着对话推进，中间内容的权重被两头挤压——开头（system prompt）和最近几轮占据主要注意力，早期的任务描述、中间步骤的关键结论逐渐沦为"背景噪音"。

直接后果：模型的行为越来越像"只看最后几页的人"。它不是不能干活，而是干的活在偏离初始目标。到后期，模型可能反复读取同一个文件却不知道为什么要读，或者产出形式正确但实质偏离的结果。

#### 4.2 compact压缩（每次都是不可逆的信息丢失）

当上下文接近窗口上限时，系统会触发compact——让模型总结之前的对话，用总结替代原文来释放空间。

问题在于：**做总结的那个模型本身已经注意力稀释了**。它用一个已经"看不清全局"的状态来决定什么值得保留，结果就是总结偏向最近的局部操作，而非最初的核心意图。

这形成了恶性循环：注意力稀释导致compact质量差，差的compact导致后续步骤进一步偏离，进一步偏离加速下一次compact的到来。每次compact就像JPEG反复保存——每次丢一点，最终面目全非。

### 五、对话模式 vs Code模式的结构性差异

两种模式擅长不同的事情，在20步任务上也以不同方式失败。

#### 5.1 对话模式（claude.ai聊天窗口）

**优势**

- 人在回路中，每一步都能看到、纠正
- 适合需要判断和方向调整的任务
- 上下文中包含用户的反馈和纠正，这些本身就是锚点

**在20步任务中的失败模式**

- 每轮用户消息+模型回复都在累加上下文
- 人工纠正能短暂恢复方向，但纠正本身也会被后续内容淹没
- **上下文被移除（compact）时，连用户之前的纠正也一起丢了**
- 对话的"闲聊"部分（确认、客套、解释）浪费了大量tokens

**真实场景下的崩溃**：考虑一个典型的复杂任务——设计一套业务分析框架，包含若干条主业务逻辑线，每条线下又有几十个细分维度需要逐一展开。这种任务在对话模式下几乎必然失败：模型需要在展开第15个维度时仍然记得第1条主逻辑线的约束，但此时早期的逻辑定义已经被中间十几轮的展开过程淹没在注意力的低权重区。更糟糕的是，对话模式在长步骤下容易触发各种中断——输出被截断、连接超时、响应异常终止——每次中断都意味着上下文的断裂，而断裂后的"继续"往往已经丢失了关键的任务状态。

**适合场景**：步骤间需要人类判断的任务，5-8步以内效果最佳

#### 5.2 Code模式（Claude Code）

**优势**

- 可以用子代理隔离上下文，每个子代理独立窗口
- CLAUDE.md提供持久化的系统级锚点
- 工具调用更高效，没有对话中的"闲聊"开销
- 有compact机制兜底

**在20步任务中的失败模式**

- 子代理返回的摘要是有损的——它认为不重要的细节可能恰恰是后续步骤需要的
- compact压缩时，**已经注意力稀释的模型做出的总结质量不可靠**
- 代码类任务有明确的成功/失败信号（编译通过、测试通过），但分析类任务没有——模型不知道自己已经偏了
- 如果CLAUDE.md写得不够精确，持久化的锚点本身就是模糊的

**适合场景**：目标明确、成功标准可验证的代码任务，子代理拆分后每段3-5步

#### 5.3 两种模式在20步任务上的核心对比

| 维度 | 对话模式 | Code模式 |
| --- | --- | --- |
| 上下文利用效率 | 低（含大量对话开销） | 高（工具调用更紧凑） |
| 目标保持能力 | 依赖人工反复提醒 | 依赖CLAUDE.md + 子代理设计 |
| 失败信号 | 人能感知到偏离 | 代码任务有测试，分析任务无信号 |
| compact后果 | 用户纠正历史被压缩丢失 | 全局规划被压缩为抽象描述 |
| 死循环风险 | 较低（人会打断） | 较高（无人监督时可能循环数轮） |
| 恢复能力 | 人工介入即可纠正（短暂有效） | 需要重启子代理或手动注入上下文 |
| 20步成功率 | 低——注意力衰减 + 上下文膨胀 | 中——如果拆成子代理可以缓解 |

### 六、核心结论

#### 6.1 两种模式共同面对的不可回避的事实

**每次compact都是一次不可逆的信息丢失。** 不是"暂时收起来以后再展开"，而是"用一个已经不清醒的大脑来决定什么值得记住"。多次compact后，原始意图被反复有损压缩，就像JPEG反复保存一样——每次都丢一点，最终面目全非。

#### 6.2 按需加载 vs 预加载的选择

按需加载（Load tools when needed）在20步任务中的优势是确定的：

- 节省15k-25k的常驻空间
- 推迟第一次compact的触发时机
- 多出的2-3步"清醒窗口"在复杂任务中可能是决定性的

#### 6.3 真正的解法不在模式选择，在任务架构

无论选哪种模式，20步连续任务的正确打开方式是**不要做20步连续任务**：

- **拆分**：把20步拆成4个5步的子任务，每个子任务独立上下文
- **外化**：每个子任务的结论写入文件（而非留在上下文中），下一个子任务从文件读取
- **锚定**：核心目标写在每个子任务的prompt开头，而非依赖模型"记住"
- **确定性压缩**：用代码逻辑（正则、字段提取）压缩中间结果，不让LLM自己总结自己

**永远不要信任模型的长期记忆。** 把目标锚定的责任从模型转移到代码层——这不是权宜之计，而是在当前架构下唯一可靠的做法。

**上下文窗口是工作台，不是仓库。工作台的正确用法是：用完清空，下一轮只摆需要的东西。**

#### 6.4 更大的图景：当前Agent的真实能力边界

上面的分析揭示了一个更深层的事实：**在现有的多步骤任务上再叠加一条新的主逻辑线，本质上依然是系统工程问题，而非AI能力问题。**

确实存在新的架构方案——比如谷歌Titans在模型内部引入了推理时动态更新的记忆模块，理论上能让模型在处理超长序列时保持对关键信息的选择性记忆。但截至目前，这类架构级的"内部记忆"还远未大规模商用落地。

这意味着什么？意味着当前市面上所有的AI Agent——无论品牌、无论宣传——在面对连续且不可中断的复杂任务时，底层都在做同一件事：**用工程手段制造"看起来智能"的效果**。子代理拆分、进度文件、compact压缩、CLAUDE.md锚定……这些都是聪明的工程，但它们是**补偿机制，不是解决方案**。

真正的检验标准很简单：给任何一个Agent一项需要20步连续推理、中间不可拆分、前后逻辑强耦合的任务——比如一个包含多条主业务逻辑线且维度间相互约束的复杂框架设计——看它能不能不丢失、不偏离、不死循环地走到终点。

截至今天，答案是：不能。

**不是模型不够聪明，而是当前的Transformer架构从根本上不具备"在工作中持续积累并精确回溯"的能力。** 所有的"记住"都是在有限窗口内的注意力分配博弈，所有的"持续"都是工程层面的断点续传。

认清这个边界，不是悲观，而是务实。知道了边界在哪，才能把精力花在有效的策略上——拆分任务、外化状态、代码层锚定——而不是反复撞墙后怀疑自己的提示词写得不够好。

### 七、一张图总结

```
步骤 1-5     步骤 6-10      步骤 11-15       步骤 16-20
[清醒期]     [漂移期]       [compact触发]     [崩溃期]
                                  
  ████████    ██████░░       ████░░░░         ██░░░░░░
  目标清晰    开始偏离       压缩丢失细节      死循环或偏离
                                  
  两种模式    两种模式       按需加载多撑      两种模式
  差异不大    差异不大       2-3步 ←关键差异   同样崩溃
```

**最终判断**：对话模式适合需要人类判断的短任务；Code模式适合可拆分、有明确验证标准的代码任务。但对于20步不可拆分的连续分析任务——两者都不够用，必须在任务架构层面解决，而非寄希望于模型层面。
