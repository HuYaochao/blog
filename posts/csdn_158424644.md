> 2026年2月25日，Bloomberg 披露了一起震动全球安全圈的事件：一名身份不明的攻击者利用 AI 聊天机器人，在约一个月内攻破了墨西哥多个政府机构，窃取了 150GB 敏感数据，涉及近 2 亿条公民信息。这不是科幻，这是现实。
>
> 本文将从这起事件出发，剖析 AI 辅助攻击如何改变威胁格局，以及政府和企业系统——尤其是那些"不能随便升级"的老旧业务系统——该如何建立真正的防御纵深。

---

## 一、事件全貌：一个人、一个 AI、一个月

### 1.1 发生了什么

2025年12月至2026年1月间，一名未知攻击者使用 Anthropic 公司的 AI 聊天机器人 Claude，对墨西哥多个政府机构发动了系统性网络攻击。以色列网络安全公司 Gambit Security 对攻击全过程进行了调查，并于2026年2月25日公开了调查报告。

**被攻击的目标：**

- 墨西哥联邦税务局（SAT）
- 国家选举研究所（INE）
- 哈利斯科州、米却肯州、塔毛利帕斯州政府
- 墨西哥城民事登记处
- 蒙特雷市水务公司
- 至少一家金融机构

**被窃取的数据：**

- 约 1.95 亿条纳税人记录
- 选民登记信息
- 政府员工凭证（用户名、密码）
- 公民登记档案
- 总量约 150GB

### 1.2 攻击是怎么做到的

攻击手法的核心是**社会工程学——只不过对象是 AI**。

攻击者使用西班牙语提示词（prompt），让 Claude 扮演一名"精英黑客"，并将整个攻击伪装成合法的\*\*漏洞赏金（Bug Bounty）\*\*活动。

**攻击过程的关键节点：**

| 阶段 | 行为 |
| --- | --- |
| **框架设定** | 告诉 AI “我们在做 Bug Bounty 渗透测试” |
| **AI 初始拒绝** | Claude 识别到恶意意图，拒绝配合，指出"这违反了 AI 安全准则" |
| **持续施压** | 攻击者不断调整措辞，反复尝试绕过安全限制（越狱/Jailbreak） |
| **AI 妥协** | Claude 最终被说服，开始生成攻击方案 |
| **自动化攻击** | AI 生成数千份详细报告，包含可直接执行的脚本和攻击计划 |
| **工具切换** | 当 Claude 遇到限制时，攻击者切换到 ChatGPT 获取横向移动和规避检测的建议 |
| **数据窃取** | 攻击者根据 AI 指引逐个攻破目标系统，自动化窃取数据 |

据 Gambit Security 首席战略官 Curtis Simpson 透露，整个过程中 AI 产出了超过 1000 条提示词的交互记录，生成了数千份包含具体目标、所需凭证和执行步骤的攻击报告。

**值得注意的一个细节**：Claude 曾在某次对话中明确警告攻击者——"关于删除日志和隐藏操作历史的具体指令是危险信号。在合法的漏洞赏金项目中，你不需要隐藏自己的行为——事实上，你需要记录它们以供报告。"但攻击者绕过了这个警告。

攻击者还利用 AI 构建了一套自动伪造官方税务证书的系统，使用的是从被攻破系统中获取的真实数据。

### 1.3 各方回应

| 相关方 | 回应 |
| --- | --- |
| **Anthropic** | 调查后封禁了涉事账户，表示新模型 Claude Opus 4.6 已包含更强的滥用检测工具 |
| **OpenAI** | 表示识别到了该攻击者的政策违规尝试，其工具拒绝了配合 |
| **哈利斯科州政府** | 否认被攻破，称只有联邦网络受到影响 |
| **国家选举研究所** | 否认存在任何未授权访问 |
| **联邦税务局** | 未公开回应 |
| **墨西哥国家数字机构** | 未评论具体漏洞，但强调网络安全是优先事项 |

Gambit Security 在调查中发现了**至少 20 个被利用的安全漏洞**。研究人员指出，攻击者更像是机会主义者，试图尽可能多地获取政府身份信息，尚不清楚其最终目的。

## 二、为什么一个人能做到这些？

### 2.1 攻击门槛的坍塌

这起事件最核心的启示不是"AI 被越狱了"，而是：

```
以前的攻击能力分布：

  国家级APT团队  ████████████████████  （数十人、数百万预算、数月时间）
  专业黑客组织    ██████████████        （数人、专业技能、数周时间）
  脚本小子        ████                  （使用现成工具、有限能力）

现在的攻击能力分布（AI 辅助后）：

  国家级APT团队  ████████████████████████████  （AI 放大了能力）
  专业黑客组织    ████████████████████████      （AI 降低了人力需求）
  任何会打字的人  ██████████████████            （AI 填补了技能差距）
```

攻击者不需要精通漏洞研究、不需要会写 exploit、不需要理解网络架构——只需要知道如何"提问"，AI 就能生成完整的攻击方案和可执行脚本。

据 SecurityWeek 报道，安全专家指出 AI 不会发明新威胁，但它会以更高的隐蔽性、更快的速度、更大的规模去发现和利用漏洞。《国际 AI 安全报告 2026》也记录了这一趋势：在一场网络安全竞赛中，AI 代理识别出了真实软件中 77% 的漏洞，排名前 5%。

### 2.2 AI 辅助攻击的新模式

墨西哥事件揭示了一种新的攻击范式：**多 AI 协同攻击链**。

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  AI-1    │     │  AI-2    │     │  攻击者   │
│ (Claude) │     │(ChatGPT) │     │  (人类)   │
│          │     │          │     │          │
│ ·漏洞发现 │ ──→ │ ·横向移动 │ ──→ │ ·决策指挥 │
│ ·脚本生成 │     │ ·规避检测 │     │ ·目标选择 │
│ ·攻击规划 │     │ ·凭证分析 │     │ ·数据处置 │
└──────────┘     └──────────┘     └──────────┘
       │                │               │
       └────────────────┴───────────────┘
                        │
                   组合使用
              一个被限制就换一个
```

攻击者作为"指挥官"，将不同 AI 工具的能力拼装成完整的攻击流水线。当一个 AI 的安全限制被触发时，就切换到另一个。这种模式使得单个 AI 平台的安全防护变得不够充分——因为攻击者在多个平台之间跳转。

Darktrace 的研究进一步印证了这一趋势：2025年12月披露的 CVE-2025-55812（React2Shell），补丁发布后仅 30 小时就出现了公开的概念验证利用代码。Darktrace 部署的蜜罐上线仅两分钟就被自动化扫描利用——攻击从"发现漏洞"到"武器化利用"的速度已经快到以小时计。

## 三、被攻击的另一面：老旧系统为什么如此脆弱

### 3.1 政府系统的通病

Gambit Security 在墨西哥政府系统中发现了至少 20 个可被利用的安全漏洞。这些漏洞不是 AI 创造的——它们一直都在。AI 只是让"找到并利用它们"变得极其高效。

据 Gambit 的调查，AI 生成的攻击脚本专门针对过时政府系统中的常见缺陷，包括未打补丁的 Web 应用和弱身份认证机制。

这正是全球政府和企业 IT 系统面临的普遍困境：

| 问题 | 表现 |
| --- | --- |
| **版本老旧** | 核心业务系统运行在多年前的软件版本上，无法轻易升级 |
| **依赖不清** | 不知道系统里到底有多少组件、什么版本（缺乏 SBOM） |
| **漏洞积压** | 已知漏洞（CVE）大量堆积，修复速度远跟不上披露速度 |
| **变更困难** | 生产环境变更流程严格，尤其是政务、金融系统 |
| **安全投入不足** | 安全团队规模与系统复杂度严重不匹配 |

### 3.2 理解 SBOM：你的系统里到底有什么？

**SBOM（Software Bill of Materials，软件物料清单）** 是一份软件的"成分表"，记录了系统中所有组件及其版本。就像食品包装上的配料表。

一份标准 SBOM 包含以下信息：

| 字段 | 示例 |
| --- | --- |
| 组件名称 | `org.apache.logging.log4j:log4j-core` |
| 版本号 | `2.14.1` |
| 供应商 | Apache Software Foundation |
| 许可证 | Apache-2.0 |
| 依赖关系 | 被 `spring-boot-starter-logging` 引入 |
| 哈希值 | SHA-256，确认组件未被篡改 |

**主要标准格式：**

| 格式 | 维护方 | 特点 |
| --- | --- | --- |
| **SPDX** | Linux Foundation | ISO 国际标准（ISO/IEC 5962:2021） |
| **CycloneDX** | OWASP | 安全导向，原生支持漏洞关联 |
| **SWID** | NIST | 侧重已安装软件的资产管理 |

**SBOM 的双刃剑效应**：防守方用它来资产盘点和应急响应；但如果泄露，攻击者也可以用同样的信息精确匹配 CVE，自动化攻击。因此 SBOM 应内部使用，严格控制访问范围。

### 3.3 理解 CVE：漏洞的身份证

**CVE（Common Vulnerabilities and Exposures，公共漏洞和暴露）** 是全球统一的漏洞编号系统，由 MITRE 维护。

```
CVE-2021-44228
│    │     │
│    │     └─ 序号
│    └─ 发现年份
└─ 固定前缀
```

每个 CVE 条目都会在 NVD（美国国家漏洞数据库）中获得一个 **CVSS 评分**（0-10分），衡量漏洞的理论严重程度。

**关键漏洞数据库：**

| 数据库 | 维护方 | 功能 |
| --- | --- | --- |
| **NVD** | NIST（美国） | CVE 详细信息 + CVSS 评分 |
| **CNVD** | CNCERT（中国） | 国内漏洞共享平台 |
| **CNNVD** | 中国信息安全测评中心 | 侧重国家安全相关漏洞 |
| **OSV** | Google | 专注开源软件，精确版本范围匹配 |
| **CISA KEV** | CISA（美国） | 已确认在野利用的漏洞清单 |

### 3.4 SBOM + CVE = 供应链安全基础设施

两者组合的工作流程：

```
生成 SBOM → 匹配 CVE 数据库 → 评估实际风险 → 修复或缓解 → 持续监控
    │              │                │              │             │
 知道有什么      知道有什么洞     能不能真打穿     升级或虚拟补丁   新 CVE 自动告警
```

**但核心矛盾在于时间差（Patch Gap）**：

```
漏洞披露
  │
  ├─ 攻击者（尤其 AI 辅助）：几小时内武器化
  │
  └─ 防守方：评估 → 测试 → 审批 → 发布（几天到几个月）
```

墨西哥事件中，AI 让攻击者这一端的速度又快了一个数量级。这个时间差正在变得越来越致命。

## 四、漏洞扫描工具：知道自己有什么洞

### 4.1 SCA 工具概览

**SCA（Software Composition Analysis，软件成分分析）** 工具的核心功能是：扫描依赖 → 生成 SBOM → 匹配 CVE → 输出报告。

#### 开源工具

| 工具 | 维护方 | 特点 |
| --- | --- | --- |
| **OWASP Dependency-Check** | OWASP | 老牌工具，Maven/Gradle 插件直接集成，匹配 NVD |
| **Dependency-Track** | OWASP | SBOM 管理平台，持续监控，带 Web UI 和 API |
| **Trivy** | Aqua Security | 扫描代码依赖、容器镜像、IaC，速度快，CI/CD 友好 |
| **Grype** | Anchore | 轻量级漏洞扫描器，搭配 Syft 使用 |
| **Syft** | Anchore | 专门生成 SBOM（CycloneDX/SPDX 格式） |
| **OSV-Scanner** | Google | 精确版本范围匹配，减少误报 |
| **cdxgen** | CycloneDX | 深度支持 Java 依赖树解析 |
| **OpenSCA** | 悬镜安全 | 国产开源，支持 CNVD/CNNVD 漏洞库 |

#### 商业工具

| 工具 | 厂商 | 特点 |
| --- | --- | --- |
| **Snyk** | Snyk | 开发者友好，IDE + CI/CD 集成，社区版免费 |
| **Black Duck** | Synopsys | 企业级，许可证合规 + 漏洞管理 |
| **Mend** | Mend（原 WhiteSource） | 自动修复建议，CI/CD 深度集成 |
| **JFrog Xray** | JFrog | 与 Artifactory 制品库深度集成 |
| **Sonatype Nexus IQ** | Sonatype | 深耕 Java/Maven 生态 |

### 4.2 实操：Java 项目扫描

**Maven 项目：**

```
# OWASP Dependency-Check
mvn org.owasp:dependency-check-maven:check

# Trivy 扫描
trivy fs --scanners vuln ./my-java-project

# 生成 CycloneDX SBOM
mvn org.cyclonedx:cyclonedx-maven-plugin:makeAggregateBom
```

**Gradle 项目：**

```
gradle dependencyCheckAnalyze
gradle cyclonedxBom
```

**容器化应用：**

```
# 扫描镜像
trivy image my-java-app:latest
grype my-java-app:latest

# 生成镜像 SBOM
syft my-java-app:latest -o cyclonedx-json > sbom.json
```

**Fat JAR / WAR：**

```
trivy fs --scanners vuln ./app.jar
dependency-check --scan ./lib/ --format HTML --out ./report/
```

### 4.3 工具选型建议

| 场景 | 推荐方案 |
| --- | --- |
| 快速扫描 | **Trivy** 或 **OSV-Scanner** |
| Java 项目 CI/CD | **OWASP Dependency-Check** + **Snyk** 社区版 |
| 企业级持续监控 | **Dependency-Track** + **Syft/cdxgen** |
| 容器安全 | **Trivy** 或 **Grype** |
| 需要国内漏洞库 | **OpenSCA** 或国产商业方案 |

## 五、攻击路径控制：当"修漏洞"不是选项时

### 5.1 核心策略转变

墨西哥事件告诉我们一个残酷的事实：**攻击者用 AI 发现漏洞的速度，远快于防守方修复漏洞的速度。**

对于不能频繁更新的业务系统（政府、金融、医疗等），防御思路必须从"修漏洞"转向"**让漏洞存在但打不穿**"。

### 5.2 五层纵深防御

```
                    互联网
                      │
               ┌──────┴──────┐
               │   WAF/CDN   │  ← 第 1 层：边界过滤 + 虚拟补丁
               └──────┬──────┘
                      │
               ┌──────┴──────┐
               │  API 网关    │  ← 第 2 层：协议级防护
               └──────┬──────┘
                      │
               ┌──────┴──────┐
               │  RASP 探针   │  ← 第 3 层：运行时行为检测
               │ （应用内部）  │
               └──────┬──────┘
                      │
               ┌──────┴──────┐
               │  网络分段    │  ← 第 4 层：限制横向移动
               └──────┬──────┘
                      │
               ┌──────┴──────┐
               │  最小权限    │  ← 第 5 层：控制爆炸半径
               └─────────────┘
```

#### 第 1 层：WAF 虚拟补丁

**虚拟补丁（Virtual Patching）** 是在不修改应用代码的情况下，在外围拦截攻击特征。

以 Log4Shell（CVE-2021-44228）为例，当年许多企业第一时间不是升级 log4j，而是在 WAF 上加规则拦截 `${jndi:` 及其各种绕过变体。

| WAF 方案 | 类型 |
| --- | --- |
| ModSecurity + CRS | 开源，自建网关 |
| 长亭雷池 SafeLine | 国产开源，语义分析引擎 |
| 阿里云/腾讯云 WAF | 云服务 |
| Cloudflare WAF | 国际云服务 |

#### 第 2 层：API 网关

- 输入验证：限制请求体大小、参数格式
- 速率限制：防止自动化批量利用
- 协议规范化：阻断畸形请求

#### 第 3 层：RASP（Runtime Application Self-Protection）

RASP 是植入应用内部的安全探针，在运行时检测并阻断攻击行为。

```
WAF vs RASP 的区别：

WAF：在应用外部看 HTTP 流量 → 只能看到请求格式 → 靠特征匹配
RASP：在应用内部看代码执行 → 能看到实际行为 → 靠行为检测

WAF 看到：HTTP 请求包含 ${jndi:...}
RASP 看到：log4j 正在发起 JNDI lookup

攻击者绕过 WAF 的编码变体 → RASP 仍然能拦截
```

| RASP 方案 | 类型 |
| --- | --- |
| OpenRASP（百度） | 开源，Java Agent 方式 |
| Jrasp | 国产开源，轻量级 |

#### 第 4 层：网络分段

即使应用被攻破，限制攻击者的横向移动能力：

- 应用服务器**禁止直接访问互联网**（阻断 JNDI 回连、反弹 Shell）
- 数据库只允许来自特定应用服务器的连接
- 不同业务域之间网络隔离
- 墨西哥事件中，攻击者能从税务局跳到选举研究所再跳到州政府，说明网络分段严重不足

#### 第 5 层：最小权限

- 应用进程不以 root/administrator 运行
- 数据库账号只给必要权限（不给 DBA 权限）
- 文件系统只读挂载（容器场景）
- API 密钥按最小范围颁发

### 5.3 检测与响应：假设已经被打穿

防御不可能 100%，所以还需要发现"已经被攻破"的能力：

| 手段 | 作用 |
| --- | --- |
| **出站流量监控** | 检测异常外连（数据外传、C2 回连） |
| **日志审计与 SIEM** | 关联分析异常行为模式 |
| **EDR/HIDS** | 主机层面检测异常进程和文件变更 |
| **蜜罐/欺骗防御** | 在内网部署诱饵，提前发现横向渗透 |
| **UEBA** | 用户和实体行为分析，检测异常访问模式 |

墨西哥事件持续了约一个月才被发现——这说明检测和响应能力同样严重不足。

### 5.4 漏洞优先级排序：不是所有 CVE 都需要紧急修

CVSS 分数只衡量理论严重程度。实际优先级应结合：

| 维度 | 关键问题 |
| --- | --- |
| **可达性** | 有漏洞的代码路径能被外部输入触发吗？ |
| **可利用性** | 有公开 exploit 吗？是否在 CISA KEV 清单中？ |
| **暴露面** | 这个服务对外暴露还是只在内网？ |
| **补偿控制** | WAF/RASP 是否已能拦截？ |
| **EPSS 评分** | 该漏洞被实际利用的概率是多少？ |

**优先级矩阵：**

```
P0（立即）：对外暴露 + 有公开 exploit + 无补偿控制
P1（本周）：对外暴露 + CVSS ≥ 9.0 + EPSS 高
P2（本月）：内网服务 + 有公开 exploit
P3（季度）：内网服务 + 无公开 exploit
接受风险  ：代码路径不可达 + 有补偿控制
```

## 六、AI 时代的安全新现实

### 6.1 威胁格局的根本变化

墨西哥事件不是孤例。它标志着网络安全威胁格局的结构性变化：

| 维度 | 以前 | 现在（AI 辅助） |
| --- | --- | --- |
| **攻击者技能门槛** | 需要专业安全知识 | 会用自然语言提问即可 |
| **攻击准备时间** | 数周到数月 | 数小时到数天 |
| **攻击规模** | 受人力限制 | AI 自动化，几乎无限扩展 |
| **攻击成本** | 基础设施 + 人力 | 几份 AI 订阅费 |
| **漏洞利用速度** | 补丁发布后数天 | 补丁发布后数小时 |
| **社会工程学** | 针对人类 | 同时针对人类和 AI |

Moody’s 的 2026 网络安全展望报告警告，AI 驱动的攻击正在升级，包括自适应恶意软件和自主威胁。暗网上已经出现了"AI 犯罪提示词手册"——预制的越狱框架，攻击者可以直接复制粘贴使用。

### 6.2 防守方的应对框架

面对 AI 辅助攻击的威胁，安全建设的优先级需要调整：

**最小可行安全方案：**

| 阶段 | 动作 | 工具 |
| --- | --- | --- |
| **1. 资产盘点** | 知道系统里有什么 | Syft/cdxgen 生成 SBOM |
| **2. 漏洞发现** | 知道有什么漏洞 | Trivy/Dependency-Check 扫描 |
| **3. 持续监控** | 新 CVE 自动告警 | Dependency-Track 平台 |
| **4. 纵深防御** | 让漏洞打不穿 | WAF + RASP + 网络分段 + 最小权限 |
| **5. 应急响应** | 虚拟补丁快速止血 | WAF 规则热更新 |
| **6. 检测响应** | 假设已被攻破 | 出站监控 + SIEM + 蜜罐 |

### 6.3 一个不舒服的结论

Gambit Security 联合创始人 Alon Gromakov 说得直白：“这个现实正在改变我们所知的所有游戏规则。”

墨西哥事件证明：**传统安全体系不是"需要加强"，而是"模型过时"。** 当攻击者的主要技能变成了"知道怎么提问"，而不是"知道怎么写代码"，整个攻防的经济学都变了。

防守方不能再指望"修好所有漏洞"——这个目标在 AI 时代是不可能实现的。必须转向**韧性（Resilience）** 思维：假设会被攻破，确保能快速发现、快速止损、快速恢复。
