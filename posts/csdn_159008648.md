> **摘要**：当前主流 LLM 服务商正在部署输出水印（output watermarking）和输出扰动（output perturbation）技术来对抗模型蒸馏攻击。本文从技术层面梳理水印嵌入的核心机制、已知的攻击手段、以及防御侧的应对演化，探讨这场攻防博弈的技术边界与内在张力。

> **导读**：整体结构从三种水印嵌入范式（绿/红名单、锦标赛采样、logits 扰动）出发，然后梳理了五种主要攻击手段（转述、token 替换、重采样、训练稀释、对抗性 prompt），再到防御侧的演化（语义水印、多层水印、自适应强度、输出扰动），最后落在博弈的结构性分析上。核心论点是那个根本矛盾：**水印信号嵌入在 token 选择层，而蒸馏提取的知识在语义层——两者抽象层次不同，这决定了水印作为反蒸馏屏障的天花板。**

---

### 一、问题定义：为什么输出水印成为必要

#### 1.1 蒸馏攻击的本质

知识蒸馏（Knowledge Distillation）原本是一种合法的模型压缩技术：用大型"教师模型"的输出来训练小型"学生模型"。当这一技术被用于未经授权地从商业 API 中提取能力时，就构成了蒸馏攻击。

攻击者的工作流程：

```
精心设计的 Prompt → 目标 API → 收集输出 → 作为训练数据 → 训练竞品模型
```

这里的关键不对称性在于：**攻击流量与正常使用在协议层完全一致**。每一条 API 请求都是语法合法、身份认证有效的——蒸馏攻击利用的不是系统漏洞，而是产品本身的功能接口。

#### 1.2 水印的目标

输出水印试图同时实现两个目标：

| 目标 | 说明 |
| --- | --- |
| **溯源（Traceability）** | 检测下游模型是否使用了自己的输出作为训练数据 |
| **降效（Degradation）** | 使输出中嵌入的统计偏差在训练过程中引入噪声，降低蒸馏效果 |

这两个目标之间存在内在张力——后文将详细讨论。

### 二、水印嵌入：三种核心范式

LLM 文本生成的基本流程是：对每个位置，模型计算词表上的概率分布（logits），然后从中采样选出下一个 token。水印技术在这条流水线的不同位置介入。

#### 2.1 范式一：采样阶段偏置（Sampling-time Biasing）

**代表方法**：KGW（Kirchenbauer et al., 2023）

**核心机制**：

1. 在每一步生成前，用一个**秘密密钥** kkk 和前 nnn 个已生成的 token 作为种子，通过哈希函数将词表 VVV 划分为"绿组"（green list）和"红组"（red list）
2. 对绿组 token 的 logits 施加一个正偏移量 δ\deltaδ（称为 hardness parameter），然后进行正常的 softmax 采样
3. 检测时，利用密钥重新计算每个位置的绿/红划分，统计文本中绿色 token 的比例，用 z-score 检验判定水印是否存在

**形式化表示**：

```
原始采样概率:  P(x_t | x_{<t})
水印采样概率:  P'(x_t | x_{<t}) ∝ exp(logit(x_t) + δ · 𝟙[x_t ∈ Green(h(x_{t-n:t-1}, k))])
```

**质量保持的直觉**：当模型对某个 token 已经非常确定时（高置信度），偏置几乎不改变结果——那个 token 无论加不加 δ\deltaδ 都会被选中。只有在多个选项概率相近时（高熵位置），偏置才实质性地影响选择，而这恰恰是人类不敏感的区域。

**水印强度的关键参数**：

- δ\deltaδ（偏移量）：越大水印越强，但文本质量下降越明显
- nnn（上下文窗口）：影响绿/红划分的随机性和可预测性
- γ\gammaγ（绿组比例）：通常设为 0.5，即词表对半分

#### 2.2 范式二：锦标赛采样（Tournament Sampling）

**代表方法**：SynthID-Text（Google DeepMind, 2024）

SynthID 采用了一种更精细的方式来嵌入水印，避免直接修改 logits：

1. 对每个位置，从原始概率分布中**重复采样多个候选 token**
2. 使用多个由密钥控制的评分函数（scoring functions）g1,g2,...,gmg\_1, g\_2, ..., g\_mg1​,g2​,...,gm​ 对候选 token 进行评分
3. 通过锦标赛规则（pairwise comparison）选出最终 token

由于候选 token 都是从**原始分布**中采样的，当某个 token 概率极高时，所有候选都是它，锦标赛结果不受影响。这使得 SynthID 在理论上是**近似无偏**的——分布偏移极小。

检测方面，SynthID 使用密钥重新计算评分函数，对文本中的 token 序列进行统计检验。其优势在于同时使用多个独立的评分函数，提供更鲁棒的检测信号。

#### 2.3 范式三：Logits 扰动（Logits Perturbation）

**代表方法**：高斯混合扰动（Fairoze et al., 2023; 以及后续工作）

直接向 logits 向量添加可控的随机扰动：

```
watermarked_logits = original_logits + ε,  ε ~ GMM(μ, σ, k)
```

其中扰动向量 ϵ\epsilonϵ 由密钥控制的高斯混合模型生成。检测器知道密钥，可以还原扰动模式并进行统计检验。

这种方法的特点是数学上更容易分析，但扰动直接作用于概率分布，对文本质量的影响需要更精细的控制。

#### 2.4 三种范式的对比

| 维度 | 绿/红名单 | 锦标赛采样 | Logits 扰动 |
| --- | --- | --- | --- |
| 嵌入位置 | logits → softmax | 采样过程 | logits |
| 分布偏移 | 有偏（biased） | 近似无偏 | 有偏 |
| 文本质量影响 | 中等 | 较小 | 取决于扰动幅度 |
| 检测灵敏度 | 高（~200 token） | 中（需更长文本） | 中 |
| 实现复杂度 | 低 | 中 | 低 |
| 生产部署 | 学术验证为主 | Google 已部署 | 学术验证为主 |

### 三、攻击侧：水印去除的技术手段

水印的安全性假设是：**攻击者不知道密钥**。在这个前提下，攻击者仍然有多种手段尝试消除或削弱水印信号。

#### 3.1 转述攻击（Paraphrase Attack）

**原理**：用另一个 LLM 对水印文本进行改写，保留语义但替换具体用词。由于水印信号依赖于具体的 token 序列，转述后的文本中绿色 token 的统计偏置会被打散。

**攻击流程**：

```
目标 API 输出（含水印）→ 转述 LLM → 去水印文本 → 用于训练
```

**效果**：这是目前最有效的水印去除手段之一。研究表明，使用 DIPPER（一种专门的转述模型）或通用 LLM 进行转述后，大多数水印方案的检测率显著下降。

**防御侧的困境**：转述攻击利用了一个根本性矛盾——水印嵌入在**表层 token 选择**中，而蒸馏训练学习的是**深层语义和推理模式**。转述改变了前者，保留了后者。

#### 3.2 Token 替换攻击（Token Substitution）

**原理**：系统性地将文本中的部分 token 替换为近义词或等价表达，破坏 token 级别的统计模式。

相比转述攻击，这种方法更轻量但也更粗糙。替换过多会影响语义质量，替换过少则水印信号残留。

#### 3.3 重采样攻击（Re-sampling Attack）

**原理**：将水印文本作为 prompt 输入到自己控制的模型中，让模型基于相同语义重新生成文本。

这本质上是一种"洗钱"操作——通过自己的模型"洗"掉原始输出中的水印特征。其效果取决于自有模型的能力：模型越强，语义保留越完整。

#### 3.4 混合训练稀释（Training Data Dilution）

**原理**：不去除单条水印，而是在训练数据中混入大量无水印数据（来自开源模型、自有数据等），通过比例稀释来降低水印信号在训练后模型中的残留。

**关键参数**：水印数据占总训练数据的比例。研究表明，当水印数据比例低于 ~10% 时，水印信号在训练后的模型中基本不可检测。

#### 3.5 对抗性提示工程（Adversarial Prompting）

**原理**：精心设计 prompt 来迫使模型进入低熵输出模式（即每个位置只有一个高概率 token），在这种情况下水印偏置无法生效。

典型策略：

- 要求模型输出结构化数据（JSON、代码）——这些场景中 token 选择的自由度低
- 要求逐字翻译或精确复述——压缩采样空间
- 要求列出具体事实、数字、名称——确定性输出居多

这种攻击利用了水印的根本局限：**水印只能嵌入在模型"犹豫"的位置，确定性输出无法承载水印信号**。

### 四、防御侧的演化

#### 4.1 鲁棒水印设计

针对转述攻击，研究者提出了**语义不变水印**（Semantic Invariant Watermark）：不在 token 层面嵌入水印，而是在更高的语义层面嵌入。

**基本思路**：

```
传统水印:  token 层面 → 转述即失效
语义水印:  句子嵌入层面 → 转述后语义不变 → 水印保留
```

具体做法包括：在句子的语义向量空间中嵌入水印信号，使得只要语义保持不变，水印就可检测。但这类方法目前仍处于早期研究阶段，检测精度和误报率尚未达到生产标准。

#### 4.2 多层水印（Layered Watermarking）

不依赖单一水印机制，而是同时部署多种不同粒度的水印：

| 层次 | 粒度 | 抗攻击特性 |
| --- | --- | --- |
| Token 级 | 单个 token 的选择偏置 | 检测灵敏度高，但不耐转述 |
| 短语级 | n-gram 模式 | 中等鲁棒性 |
| 句子级 | 句法结构偏好 | 耐转述，但信号弱 |
| 风格级 | 整体文本风格指纹 | 最鲁棒，但最难检测 |

攻击者需要同时破解所有层次，防御侧只需要任一层次仍然有效即可检测。这构成了一种**不对称防御**。

#### 4.3 自适应水印强度

根据请求的可疑程度动态调整水印强度：

```
正常用户:  低强度水印 → 质量几乎无损
可疑流量:  高强度水印 → 质量略降但信号强
确认蒸馏:  最高强度 + 选择性降质 → 显著降低蒸馏价值
```

这需要配合流量分析系统（行为指纹、多账户协同检测等）来判断可疑程度。**水印不再是静态的被动标记，而是成为动态响应系统的一部分**。

#### 4.4 输出扰动（Output Perturbation）

严格来说，输出扰动和水印是两种不同的技术，但常被混为一谈：

| 维度 | 水印 | 输出扰动 |
| --- | --- | --- |
| 主要目标 | 事后检测/溯源 | 事前降低蒸馏效果 |
| 信号可逆性 | 嵌入可检测的信号 | 引入不可逆的噪声 |
| 对正常用户的影响 | 设计上无感知 | 可能存在微小影响 |
| 对蒸馏的影响 | 间接（通过分布偏移） | 直接（训练数据质量下降） |

输出扰动的核心思路：在 token 概率空间中引入**对人类无感但对梯度信号有害**的扰动。人类阅读文本时关注语义，容忍 token 级别的变化；而蒸馏训练依赖 token 级别的精确分布，扰动会直接污染梯度信号。

### 五、博弈的结构性分析

#### 5.1 攻防不对称性

这场博弈存在一个根本的结构性不对称：

```
防御侧的目标:  在 token 层面嵌入信号，使之同时满足——
                ① 人类不可感知
                ② 统计上可检测
                ③ 抗各种去除攻击

攻击侧的目标:  保留语义层面的知识，丢弃 token 层面的信号
```

**问题在于：蒸馏学习的是语义和推理模式，而水印嵌入在 token 选择中。这两者处于不同的抽象层次。**

攻击者不需要精确复制 token 序列——只需要从中提取知识。这意味着任何只在 token 层面工作的水印，理论上都可以通过足够好的语义提取来绕过。

#### 5.2 信息论视角

从信息论的角度看，水印面临一个根本约束：

**水印容量受限于采样熵**。

在模型高度确定的位置（低熵），水印无法嵌入任何信息。只有在模型"犹豫"的位置（高熵），水印才有嵌入空间。而高质量的 LLM 在很多任务上趋向低熵输出——这意味着**模型越好，可嵌入水印的空间越小**。

形式化地说：

```
水印可嵌入信息量 ≤ Σ_t H(P(x_t | x_{<t}))   (各位置的条件熵之和)
```

对于代码生成、数学推理、事实问答等低熵任务，这个上界很低。

#### 5.3 当前均衡点

综合攻防双方的能力，当前的技术均衡大致是：

| 场景 | 水印效果 | 攻击者绕过难度 |
| --- | --- | --- |
| 原始文本溯源（未经处理） | 非常有效 | 不适用 |
| 轻度转述后检测 | 部分有效 | 低 |
| 深度转述/重写后检测 | 基本失效 | 中 |
| 蒸馏后模型中的水印残留检测 | 效果有限 | 中-高 |
| 大规模混合训练后的检测 | 接近无效 | 低 |

**关键洞察**：水印在**溯源**场景（证明某段文本来自特定 API）上相当有效，但在**防蒸馏**场景（阻止攻击者从输出中提取知识）上效果有限。这两个目标需要不同的技术路线。

#### 5.4 更深层的博弈：水印之外

正因为 token 级水印对蒸馏防御的局限性，实际部署的反蒸馏系统采用的是**多层防御策略**，水印只是其中一环：

```
第一层: 流量检测（行为指纹、异常模式识别）     → 发现攻击
第二层: 访问控制（身份验证、区域限制、速率限制）  → 阻断攻击
第三层: 输出水印 + 扰动                        → 降低攻击价值
第四层: 模型层面防御（推理路径混淆等）           → 最后一道屏障
```

水印更多承担的是**威慑**和**取证**功能——让攻击者知道"你的行为可以被追踪"，并在法律/商业层面提供证据——而非技术上"不可能蒸馏"。

### 六、开放问题与未来方向

#### 6.1 无偏水印与检测灵敏度的张力

无偏水印（如 SynthID 的锦标赛采样）保持了输出分布不变，代价是信号强度降低，需要更长的文本才能检测。有偏水印（如 KGW）检测灵敏度高，但会微妙地改变输出分布。

**是否存在一种水印同时满足无偏性和高检测灵敏度？** 信息论分析表明这两者之间存在根本性的 trade-off，但精确的帕累托边界尚未完全刻画。

#### 6.2 对抗鲁棒性的理论上限

**在保持语义不变的前提下，水印的最大鲁棒性是多少？** 如果攻击者有一个完美的语义提取器（保留全部语义但重写所有 token），那么任何 token 级水印都会失效。这意味着水印鲁棒性的上限由"语义等价类"的大小决定——同一语义可以用多少种不同的方式表达。

#### 6.3 蒸馏检测的替代路径

一些研究开始探索不依赖水印的蒸馏检测方法：

- **模型指纹（Model Fingerprinting）**：在训练中植入特定的"后门行为"——对特定 trigger 输入给出特定输出。如果下游模型也表现出这些行为，则说明存在蒸馏关系
- **成员推理攻击（Membership Inference）**：检测特定样本是否被用于训练——如果对方模型在你的 API 输出上表现出异常的"熟悉度"，则可能存在蒸馏
- **行为克隆检测**：分析两个模型在特定输入上的错误模式是否异常相关——独立训练的模型不太可能犯完全相同的错误

#### 6.4 动态博弈与军备竞赛

水印技术的演化遵循典型的安全军备竞赛模式：

```
水印 v1 → 转述攻击 → 语义水印 → 更强的转述器 → 多层水印 → 混合攻击 → ...
```

每一代防御技术都会激发新的攻击方法，反之亦然。长期来看，这场博弈的均衡点可能不取决于技术本身，而取决于攻防双方的成本比——当攻击成本高到不如自己训练时，蒸馏攻击自然停止。

---

### 七、结论

LLM 输出水印是一个技术上精巧但边界明确的工具。它在溯源取证方面已经具备实用价值，但作为反蒸馏的技术屏障，其效果受限于一个根本矛盾：**水印信号和蒸馏所提取的知识处于不同的抽象层次**。

对从业者而言，几个关键认识：

1. **水印不是银弹**——它是多层防御中的一环，不能独立承担反蒸馏的全部重任
2. **水印的主要价值在于溯源和威慑**——而非技术上"不可能被绕过"
3. **攻防均衡由成本决定**——技术上的绝对安全不存在，但可以将攻击成本提高到经济上不合算
4. **水印设计需要针对具体威胁模型**——面对不同水平的攻击者，需要不同的水印策略

这场博弈还远未结束，但其走向已经清晰：纯技术对抗将趋向均衡，最终的解决方案将是技术、法律、商业手段的组合。
